## What is a Squad?

A squad is a team of AI agents assigned to work on a specific codebase. Each squad has:

- **Team Lead** — Coordinates work, breaks down tasks, reviews output (required, has veto power)
- **QA/Tester** — Validates work quality (required, has veto power)
- **Specialists** — Dynamic roles based on project needs (e.g., "React Expert", "API Designer")

A squad must have at minimum a Team Lead and a QA/Tester. These can be the same agent if the squad is small.

## Universe Naming

Squad members get character names from a pop-culture universe chosen during hiring. For example, with the "A-Team" universe:

| Role | Display Name | Character |
|------|-------------|-----------|
| Team Lead | Hannibal | Strategic planner who "loves it when a plan comes together" |
| React Developer | Murdock | Unpredictable creative who gets wild ideas |
| QA/Tester | B.A. Baracus | Tough critic, won't let sloppy work through |

Each agent has:
- **`display_name`** — The character name (unique within the squad)
- **`role_name`** — The technical specialty (e.g., "react-developer")
- **`persona`** — LLM-generated personality that influences communication style

The universe is chosen at squad creation and names are generated by the LLM to fit both the character's traits and the agent's role.

## Squad Lifecycle

```
1. User asks IO to "hire a squad for project X"
2. Orchestrator analyzes the repo (languages, structure, complexity)
3. Recommends team composition → user confirms
4. Squad is created with SKILL.md files for each role
5. Squad is ready to accept delegated work
```

## SKILL.md Format

Each agent's identity is defined by a SKILL.md file stored in `~/.io/squads/{squad-name}/`:

```markdown
---
role: team-lead
tools:
  - read_file
  - list_files
  - create_task
  - review_code
model: claude-opus-4.6
veto: true
---

You are the Team Lead for the {project} squad.
Your job is to break down objectives into tasks,
assign them to team members, and review their output.
```

## Instances

When a squad receives work, it creates an **instance** — an isolated execution context:

1. **Planning** — Team lead analyzes the objective
2. **Meeting** — Round-table discussion (all agents contribute, veto members can block)
3. **Working** — Tasks assigned and executed in parallel
4. **Reviewing** — Team lead reviews all output
5. **Complete** — PR created via `gh` CLI

Each instance gets its own git worktree for isolation. Max 3 concurrent instances per squad.

## Autonomy Tiers

Squads operate at configurable autonomy levels:

| Tier | Can Merge PRs | Can Release | Can Close Issues | Self-Review |
|------|:---:|:---:|:---:|:---:|
| Low | ❌ | ❌ | ❌ | ❌ |
| Medium | ❌ | ❌ | ✅ | ✅ |
| High | ✅ | ✅ | ✅ | ✅ |

## Round-Table Meetings

Before work begins, agents hold a meeting:
- Each agent contributes their perspective in round-robin order
- Multiple rounds can occur for complex decisions
- Veto members can block consensus
- The scribe records the final decision
